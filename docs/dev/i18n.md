# i18n Guide

## Overview

- Source of truth: `src/lib/shared/i18n/locales/en.json` and `src/lib/shared/i18n/locales/ru.json`.
- Supported locales: `en`, `ru`.
- Runtime: `frontend/src/lib/i18n/index.ts`.
- Locale cookie: `archimap_locale`.

## How locale is resolved

1. URL path locale segment (if present and supported).
2. Query parameter `?lang=`.
3. Cookie `archimap_locale`.
4. Browser language (`navigator.languages`).
5. Default locale (`en`).

## Transactional emails

- Registration, password reset, and SMTP test emails resolve locale from the request locale or body, then `archimap_locale`, then `Accept-Language`, then default to `en`.
- Email copy itself lives in the same locale JSON files under `email.*`, so UI strings and transactional copy share one source of truth.
- Registration and password reset links include `lang=` so the web UI opens in the same language when the recipient follows the link.
- Frontend UI and email templates both import the shared locale files through the `$shared` alias.

## Adding a new key

1. Add key to `src/lib/shared/i18n/locales/en.json`.
2. Add the same key to `src/lib/shared/i18n/locales/ru.json`.
3. Use key from UI via `$t('section.key')` in Svelte or `translateNow('section.key')` in JS.
4. Run `npm run i18n:validate`.

## Adding a new language

1. Add `<locale>.json` in `src/lib/shared/i18n/locales/`.
2. Register locale in `frontend/src/lib/i18n/config.ts` (`SUPPORTED_LOCALES`).
3. Add human-readable label under `locale.<code>` in all locale files.
4. Run validation and checks.

## Naming rules

- Use nested keys, not flat keys.
- Group by domain: `common.*`, `header.*`, `account.*`, `admin.*`, `errors.*`.
- Keep keys stable and semantic (`admin.status.pending`, not `label1`).

## Fallback behavior

- Missing key in active locale falls back to default locale (`en`).
- If key is missing in all locales, key name is returned.
- In non-production mode missing keys are logged (`[i18n] Missing key: ...`).

## Commands

- `npm run i18n:extract` - list translation keys used in code.
- `npm run i18n:validate` - compare locale keysets.
- `npm run i18n:check` - detect potential hardcoded UI strings.
