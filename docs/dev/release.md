# Release Versioning

## Source of Truth

- Release version is derived from Git tags in the `vX.Y.Z` format.
- Build metadata includes:
  - short commit hash
  - build UTC timestamp
  - dirty flag (when working tree has changes)

Generated files:

- `src/lib/version.generated.json` (backend)
- `frontend/src/lib/version.generated.json` (frontend)

Both files are generated by [`node --import tsx scripts/generate-version.ts`](../../scripts/generate-version.ts) and should not be committed.

## Release Flow (Tag-based)

1. Ensure the target commit is on `main`/release branch and CI is green.
2. Create and push tag:
   - `git tag v1.4.2`
   - `git push origin v1.4.2`
3. Build pipeline uses the tag as version source automatically.
4. Verify:
   - `GET /api/version` returns `version: "1.4.2"`
   - `/info` shows `1.4.2 (<commit>, <buildTime>)`

## Local Verification

1. Run `npm run build`.
2. Run `npm run version:print`.
3. Confirm generated JSON contains expected `version`, `git.commit`, and `buildTime`.

## Build Without `.git`

- If Git metadata is not available (for example Docker build context without `.git`), generation does not fail.
- Version falls back to `package.json` version and includes `buildTime`.
- The script logs a warning:
  - `[version] git metadata is unavailable, using package.json fallback`

## Docker Release

Build and push image via registry:

Scripts:

- [`scripts/release-docker.ps1`](../../scripts/release-docker.ps1)
- [`scripts/release-docker.sh`](../../scripts/release-docker.sh)

- `./scripts/release-docker.ps1 -Version 1.2.3`

Deploy on server via layer-based pull:

- `docker pull streletskiy/archimap:1.2.3`
- `docker compose up -d`

Do not use tar-based transport (`docker save` / `docker load`) for releases, because it forces full image transfer and bypasses efficient layer reuse.
